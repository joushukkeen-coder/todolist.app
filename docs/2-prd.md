## 1. 제품 개요

### 목적

20대~50대 직장인이 개인 할일을 안전하고 체계적으로 관리할 수 있는 인증 기반 할일 관리 웹 애플리케이션을 제공한다.

### 제품 비전

> "언제 어디서든, 내 할일을 내가 원하는 방식으로 관리한다."

반응형 웹 UI를 통해 PC와 모바일 환경 모두에서 동일한 사용자 경험을 제공하며, 카테고리 기반의 체계적 분류와 다양한 필터링으로 업무 생산성을 향상시킨다.

### 범위 요약

| 구분 | v1 (MVP) | v2 (확장) |
|------|----------|-----------|
| 인증 | JWT 기반 이메일/비밀번호 | OAuth Social (Google, Facebook 등) |
| 플랫폼 | 반응형 웹 (Web + Mobile Web) | 네이티브 앱 검토 |
| UI 테마 | 라이트 모드 단일 | 다크 모드 추가 |
| 언어 | 한국어 단일 | 다국어 지원 (i18n) |

## 2. 타겟 사용자

### 주요 사용자 페르소나

| 구분 | 내용 |
|------|------|
| 연령대 | 20대 ~ 50대 |
| 직군 | 직장인 (업종 무관) |
| 디바이스 | PC (업무 시간), 스마트폰 (이동 중) |
| 사용 목적 | 업무 및 개인 할일의 체계적 관리, 완료 현황 추적 |
| 기술 숙련도 | 일반 수준 (스마트폰 앱 및 웹 서비스에 익숙함) |

### 주요 사용 시나리오

1. **출근 전** — 오늘 처리할 업무 할일 목록을 모바일로 확인
2. **업무 중** — PC 브라우저에서 할일 등록·수정·완료 처리
3. **퇴근 후** — 카테고리/완료 여부 필터로 잔여 할일 파악

## 3. 기능 요구사항

도메인 정의서의 유스케이스를 기준으로 한다. 모든 유스케이스는 v1 MVP에 포함된다.

### 3.1 사용자 관리

| ID | 기능 | 설명 | 인수 조건 |
|----|------|------|-----------|
| US-U-01 | 회원가입 | 이메일 + 비밀번호로 계정 생성 | 이메일 중복 검증 후 계정 생성, 비밀번호 해시 저장 |
| US-U-02 | 로그인 | 이메일 + 비밀번호로 JWT 발급 | 자격 증명 일치 시 Access Token 반환, 불일치 시 401 오류 |
| US-U-03 | 프로필 수정 | 이름 또는 비밀번호 변경 | 변경 항목만 업데이트, 이메일 변경 불가 |

### 3.2 카테고리 관리

| ID | 기능 | 설명 | 인수 조건 |
|----|------|------|-----------|
| US-C-01 | 카테고리 생성 | 사용자 정의 카테고리 추가 | 이름 중복 검증 후 생성, 기본 카테고리와 함께 조회 |
| US-C-02 | 카테고리 수정 | 이름 또는 색상 코드 변경 | 기본 카테고리 수정 불가, 사용자 정의 카테고리만 허용 |
| US-C-03 | 카테고리 삭제 | 사용하지 않는 카테고리 삭제 | 연결된 할일이 없을 경우에만 삭제 허용 |

**기본 카테고리** (시스템 제공, 수정·삭제 불가)

| 이름 | 색상 코드 |
|------|-----------|
| 개인 | `#4A90D9` |
| 업무 | `#E8503A` |
| 쇼핑 | `#2ECC71` |

### 3.3 할일 관리

| ID | 기능 | 설명 | 인수 조건 |
|----|------|------|-----------|
| US-T-01 | 할일 등록 | 제목·설명·종료예정일·카테고리 지정 | 제목(1~200자)·카테고리 필수, 설명·종료예정일 선택 |
| US-T-02 | 할일 목록 조회 | 본인 소유 할일 전체 조회 | 생성일시 역순 기본 정렬, 타인 할일 접근 불가 |
| US-T-03 | 할일 수정 | 제목·설명·종료예정일·카테고리 수정 | 본인 소유 검증 후 수정 허용 |
| US-T-04 | 완료 처리 | 할일을 완료 상태로 변경 | 완료 여부 및 완료일시 업데이트 |
| US-T-05 | 완료 취소 | 완료 처리된 할일을 미완료로 복원 | 완료 여부 해제, 완료일시 초기화 |
| US-T-06 | 할일 삭제 | 할일 영구 삭제 | 본인 소유 검증 후 영구 삭제 |
| US-T-07 | 할일 필터링 | 카테고리·기간·완료 여부 복합 필터 | 복합 조건 지원, 기간 필터는 종료예정일 기준, 종료예정일 없는 할일은 기간 필터 결과에서 제외 |

## 4. 비기능 요구사항

### 4.1 성능

| 항목 | 목표값 |
|------|--------|
| 최대 동시 접속자 | 300명 |
| API 응답 시간 (p95) | 500ms 이내 |
| 페이지 초기 로딩 (FCP) | 2초 이내 |

### 4.2 가용성 및 규모

- 소규모 개인 프로젝트 수준의 인프라로 운영
- 단일 서버 배포 (수평 확장은 v2 이후 검토)

### 4.3 보안

| 항목 | 정책 |
|------|------|
| 인증 토큰 | JWT, Access Token 유효기간 1시간 |
| 토큰 저장 방식 | Zustand `authStore` 메모리에 저장. `localStorage` · `sessionStorage` · `HTTP Only Cookie` 사용 금지. 페이지 새로고침 또는 브라우저 종료 시 토큰이 초기화되며 재로그인이 필요하다 |
| 비밀번호 저장 | bcrypt 해시 (salt rounds ≥ 10), 최소 8자 이상 검증 |
| 데이터 격리 | 모든 API에서 userId 기준 소유권 검증 |
| HTTPS | 운영 환경 필수 적용 |
| SQL Injection | pg 라이브러리 파라미터 바인딩으로 방어 |

### 4.4 데이터 보존 정책

| 시나리오 | 정책 |
|----------|------|
| 회원 탈퇴 | 사용자 데이터(할일·카테고리 포함) 즉시 영구 삭제 |
| 할일 삭제 | 즉시 영구 삭제 (소프트 삭제 없음) |

### 4.5 브라우저 지원

| 브라우저 | 지원 버전 |
|----------|-----------|
| Chrome | 최신 2버전 |
| Safari | 최신 2버전 |
| Firefox | 최신 2버전 |
| Edge | 최신 2버전 |
| 모바일 Safari / Chrome | 최신 2버전 |

## 5. 기술 스택

### 5.1 프론트엔드

| 항목 | 기술 |
|------|------|
| 프레임워크 | React 19 |
| 언어 | TypeScript |
| 상태 관리 | Zustand |
| 서버 상태 | TanStack Query |
| UI | 반응형 CSS (모바일 우선 설계) |

### 5.2 백엔드

| 항목 | 기술 |
|------|------|
| 런타임 | Node.js |
| 프레임워크 | Express |
| API 방식 | REST API |
| 인증 | JWT (jsonwebtoken) |
| DB 연동 | **pg 라이브러리 (필수)** — ORM 사용 금지 |
| 비밀번호 해시 | bcrypt |

> **제약**: PostgreSQL 연동은 반드시 `pg` 라이브러리를 직접 사용한다. Sequelize, Prisma 등 ORM/Query Builder 사용 금지.

### 5.3 데이터베이스

| 항목 | 기술 |
|------|------|
| DBMS | PostgreSQL 17 |
| 주요 설계 원칙 | UUID PK, 타임스탬프 컬럼 표준화, Index 설계 |

### 5.4 인증 전략

```
v1 (MVP)
└── JWT 이메일/비밀번호 인증
    ├── POST /auth/register
    ├── POST /auth/login
    ├── Authorization: Bearer <token>
    └── 클라이언트 토큰 저장: Zustand authStore (메모리)

v2 (확장)
└── OAuth 2.0 Social 로그인
    ├── Google
    └── Facebook (및 추가 공급자)
```

## 6. 시스템 아키텍처 개요

```
[클라이언트]
 React SPA (반응형 웹)
    │  HTTP/HTTPS REST API
    ▼
[백엔드]
 Node.js + Express
 ├── Auth Router      (/api/auth)
 ├── Users Router     (/api/users)
 ├── Categories Router(/api/categories)
 └── Todos Router     (/api/todos)
    │  pg 라이브러리
    ▼
[데이터베이스]
 PostgreSQL 17
```

### API 기본 규칙

| 항목 | 규칙 |
|------|------|
| Base URL | `/api/v1` |
| 인증 헤더 | `Authorization: Bearer <JWT>` |
| 응답 형식 | JSON |
| 에러 형식 | `{ "error": { "code": "...", "message": "..." } }` |
| 날짜 형식 | ISO 8601 (`YYYY-MM-DDTHH:mm:ssZ`) |

## 9. 데이터 모델 요약

도메인 정의서의 엔티티를 기반으로 하며, PostgreSQL 테이블 설계 시 아래를 따른다.

| 테이블 | 주요 컬럼 | 비고 |
|--------|-----------|------|
| `users` | `user_id UUID PK`, `email`, `password_hash`, `name`, `created_at`, `updated_at` | 이메일 UNIQUE |
| `categories` | `category_id UUID PK`, `user_id UUID FK NULL`, `name`, `color_code`, `is_default BOOL`, `created_at` | `user_id NULL` = 기본 카테고리 |
| `todos` | `todo_id UUID PK`, `user_id UUID FK`, `category_id UUID FK`, `title`, `description`, `due_date`, `is_completed BOOL`, `completed_at`, `created_at`, `updated_at` | |

## 10. 위험 요소 및 대응

| 위험 요소 | 대응 방안 |
|-----------|-----------|
| 3일 일정 내 전체 기능 완성 불확실 | 백엔드 API 우선 완성 후 프론트엔드 순차 적용, 화면 완성도보다 기능 동작 우선 |
| JWT 탈취 | HTTPS 강제 적용, 토큰 유효기간 단축 (1시간), 향후 Refresh Token 도입 검토 |
| pg 직접 쿼리 SQL Injection | 파라미터 바인딩 (`$1, $2, ...`) 일관 적용, 동적 쿼리 생성 금지 |
| OAuth 확장 시 인증 구조 변경 | v1 설계 시 `auth_provider` 컬럼 예약, 이메일 인증을 `local` 타입으로 구조화 |
